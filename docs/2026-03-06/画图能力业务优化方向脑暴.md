# 画图能力业务优化方向脑暴（2026-03-06）

## 1. 文档目的

本文用于记录当前基于代码现状得出的业务优化方向，作为后续专项评估、需求拆解和排期讨论的输入材料。

当前内容以“画图能力”和“模块间业务闭环”为核心，不代表最终立项结论，后续仍需补充：

- 用户场景优先级评估
- 技术实现成本评估
- 与现有前后端架构的兼容性评估
- 商业化/留存/传播价值评估

---

## 2. 当前产品真实基础能力

结合当前前后端代码，项目已经具备以下基础模块：

### 2.1 双画布基础

- `ReactFlow` 结构化画布：适合节点、边、布局、结构调整、导出。
- `Excalidraw` 手绘画布：适合草图、演示表达、手绘风呈现。

这意味着产品已经天然具备“双模画图”的潜力，不需要再从零引入第三套主画布。

### 2.2 多输入源基础

当前已具备或部分具备：

- 文本生成流程图/架构图
- 图片识别成图
- 图片转 Excalidraw
- Mermaid 解析与生成
- 模型配置管理
- 会话保存与增量生成
- 导出 PPT / Slidev / 讲稿

### 2.3 闭环雏形已存在

当前已经不是一个单纯“生成图”的 Demo，而是已经初步具备：

`输入 -> 生成 -> 编辑 -> 导出`

但还没有形成真正完整的业务闭环，主要差在：

- 导入/导出链不完整
- 双画布之间还是割裂的
- 分享/复用/协作能力弱
- 模板资产与知识沉淀不足

---

## 3. 建议的业务定位

建议将本项目从“AI 画图工具”进一步升级为：

> **AI Diagram Workspace / AI 画图工作台**

核心不是“一次性生成一张图”，而是围绕图形成完整的业务链路：

- 多种输入进入
- AI 生成初稿
- 用户快速编辑修图
- 多格式导出传播
- 图转内容 / 图转汇报
- 模板沉淀 / 项目复用 / 团队协作

换句话说，目标不是只让用户“生成成功”，而是让用户“把图真正用起来”。

---

## 4. 重点业务优化方向

## 4.1 方向一：建立统一导出中心（Export Hub）

### 为什么优先级高

这是最短路径、最容易体现业务价值的一步。

当前已有：

- PPT 导出
- Slidev 导出
- 讲稿导出

但从画图产品视角，还缺少用户最常用的导出能力：

- PNG
- SVG
- ReactFlow JSON
- Mermaid
- Excalidraw JSON

### 建议补齐的导出类型

#### ReactFlow 侧

- PNG：便于飞书、钉钉、邮件、IM 场景快速传播
- SVG：适合高质量展示、网页嵌入、文档插入、打印
- JSON：用于继续编辑、版本恢复、跨环境迁移
- Mermaid：适合文档化、Git 化、工程内复用

#### Excalidraw 侧

- PNG
- SVG
- `.excalidraw` / Scene JSON

### 业务价值

- 提升“画完即带走”的完成度
- 提升分享传播能力
- 提升复用能力
- 为后续版本历史、模板库、协作打基础

### 后续评估点

- 是否做单一导出按钮，还是统一“导出中心”弹窗
- 是否区分“展示格式”和“可编辑格式”
- 导出内容是否包含标题、说明、元数据

---

## 4.2 方向二：打通 ReactFlow 与 Excalidraw 的双画布协同

### 当前问题

现有双画布能力具备，但业务上仍然偏割裂：

- ReactFlow 更像结构图编辑器
- Excalidraw 更像独立的 AI 手绘生成器

两者缺少明确的角色分工和互转路径。

### 建议定位

#### ReactFlow

用于：

- 结构化架构图
- 流程图
- 节点/边精确编辑
- 自动布局
- 可计算导出

#### Excalidraw

用于：

- 汇报态表达
- 手绘风演示
- 白板式脑暴
- 批注与讲解

### 建议新增业务按钮

- `转为 Excalidraw 演示图`
- `转为 ReactFlow 结构图`
- `同步当前图到另一画布`

### 推荐策略

不要追求“任意图完全无损互转”，而应分层处理：

- 结构化图：高保真互转
- 手绘自由图：允许降级为视觉导入，不承诺结构化还原

### 更合理的中间层

建议后续定义统一文档模型，例如：

`SmartArchitectDocument`

包含：

- nodes
- edges
- viewport
- theme/style
- annotations
- source metadata
- export metadata

让 ReactFlow / Mermaid / Excalidraw / SVG 都围绕同一份业务文档做转换。

### 业务价值

- 形成“结构图 + 演示图”双场景闭环
- 一套图可同时满足设计、评审、汇报
- 减少用户重复画图成本

---

## 4.3 方向三：引入 SVG 作为核心分发格式与交换层

### 为什么值得引入

SVG 在业务上非常适合做：

- 高质量导出
- 网页嵌入
- 文档展示
- 打印/PDF
- 作为视觉交换层

### 建议角色定义

SVG 不建议作为唯一源数据，但非常适合作为：

- **分发格式**
- **展示格式**
- **视觉中间格式**

### 可拓展的业务场景

- 在官网/产品站嵌入架构图
- 导出到设计文档 / 汇报材料
- 从 SVG 导入 Excalidraw 做演讲式再编辑
- 生成品牌化风格图

### 业务价值

- 提高图的传播质量
- 打通“画图 -> 内容资产”的链路
- 有助于与外部文档系统集成

### 后续评估点

- SVG 是否支持主题风格导出
- 导出时是否保留可选的节点元数据
- 是否支持面向网页嵌入的轻量版本

---

## 4.4 方向四：建立统一导入中心（Import Hub）

### 当前输入方式还不够完整

目前已有：

- 文本输入
- 图片上传识别

但对一个画图工作台来说，还应该补齐常见导入源：

- Mermaid
- SVG
- ReactFlow JSON
- Excalidraw JSON / Scene
- 历史模板
- 历史项目

### 建议的导入中心入口

统一入口：

- 新建空白图
- 从文本生成
- 从图片识别
- 导入 Mermaid
- 导入 SVG
- 导入 JSON
- 从模板创建
- 从历史项目继续编辑

### 业务价值

- 把产品从“生成器”变成“工作台”
- 降低用户迁移成本
- 提高回访率和复用率

---

## 4.5 方向五：补齐画图增强能力，而不仅是 AI 生成

### 核心判断

很多产品停留在“AI 出一张图”，但实际高频需求是：

> **AI 先生成，人工再快速修图**

所以画图效率不只来自 AI，也来自后续编辑体验。

### 建议增强的能力

#### ReactFlow 侧

- 对齐 / 分布 / 吸附 / 网格
- 分组 / 解组
- 锁定节点
- 快速替换节点类型
- 边样式与箭头样式快速切换
- 画布标注层
- 版本回退

#### Excalidraw 侧

- 演讲标注模式
- 重点区域高亮
- 讲解路径
- 场景式截图导出

### 业务价值

- 让用户从“能生成”走向“真能改”
- 降低 AI 生成不完美时的挫败感
- 提升专业用户使用深度

---

## 4.6 方向六：建设模板库和组件资产库

### 为什么重要

用户长期留存往往不来自“再生成一次”，而来自：

- 模板复用
- 组件复用
- 行业知识复用

### 建议分层

#### 模板库

- 流程模板
- 架构模板
- 行业模板
- 汇报模板

#### 组件库

- API Gateway
- Redis / Cache
- MQ / Kafka
- 数据库
- CDN / OSS
- 微服务
- LLM / Agent / RAG
- 监控 / 日志 / 网关 / 鉴权

#### 风格模板

- 正式汇报风
- 技术方案风
- 手绘脑暴风
- 深色演示风

### 业务价值

- 减少重复劳动
- 提升生成质量
- 强化行业场景适配
- 为后续企业版能力做准备

---

## 4.7 方向七：补齐分享、评审、协作闭环

### 当前短板

图产出了，但传播和团队协作链路偏弱。

### 建议新增能力

- 分享链接
- 只读视图
- 嵌入页面/文档
- 导出评审版图片/SVG
- 节点级评论
- 版本对比
- 审阅记录

### 业务价值

- 从个人工具向团队工具升级
- 提升传播与讨论效率
- 支撑方案评审场景

---

## 4.8 方向八：形成“图 -> 内容 -> 汇报”的上层闭环

### 当前基础很好

项目已经具备：

- 图转 PPT
- 图转 Slidev

这意味着产品有机会成为：

> 不只是画图工具，而是“方案表达工具”

- 生成“演讲模式”版本图
- 图与讲稿双向联动
- 汇报时按节点高亮路径

### 业务价值

- 直接服务售前、架构评审、汇报场景
- 放大单张图的商业价值
- 强化差异化竞争力

## 6. 建议的分阶段推进顺序

## 第一阶段：补全基础闭环

目标：让“画完图 -> 带走 -> 再回来编辑”成立。

建议优先做：

- ReactFlow 增加 PNG / SVG / JSON / Mermaid 导出
- Excalidraw 增加 SVG / JSON 导出入口梳理
- 增加导入中心（先支持 Mermaid / JSON）
- 统一导出中心 UI

## 第二阶段：打通双画布协同

目标：让 ReactFlow 和 Excalidraw 成为一套系统，而不是两个功能点。

建议优先做：

- ReactFlow -> Excalidraw
- Mermaid -> Excalidraw
- SVG -> Excalidraw
- 部分结构图 Excalidraw -> ReactFlow

## 第三阶段：提升编辑效率与专业度

目标：让产品不只会“生成”，也能“精修”。

建议优先做：

- 对齐/分布/吸附/分组
- 标注层
- 风格模板
- 版本管理

## 第四阶段：补齐协作与资产沉淀

目标：从个人画图工具升级为团队图形工作台。

建议优先做：

- 分享链接
- 只读嵌入
- 评审批注
- 模板库/组件库
- 历史项目与知识沉淀

## 7. 当前最值得优先评估的事项

# 附录 A：ReactFlow 导出中心专项评估

## A.1 评估目的

本附录用于单独评估“ReactFlow 导出中心”是否值得优先立项，以及建议做到什么深度。

核心问题不是“能不能导出”，而是：

- 导出的内容是否覆盖真实业务场景
- 导出的格式是否能支撑后续再编辑
- 导出是否能打通传播、汇报、复用三个环节

---

## A.2 当前现状

当前产品已经具备部分导出能力：

- PPT 导出
- Slidev 导出
- 讲稿导出

但从结构化画图产品视角，仍然缺少最常用的基础导出格式：

- PNG
- SVG
- ReactFlow JSON
- Mermaid

也就是说，当前导出更偏“汇报内容输出”，而不是“图形资产输出”。

---

## A.3 业务场景拆解

### 场景 1：快速传播

典型用户诉求：

- 发到飞书、微信、邮件
- 放进群里快速讨论
- 给领导或同事看一个当前版本

对应最合适的导出格式：

- PNG

### 场景 2：正式文档沉淀

典型用户诉求：

- 放到方案文档
- 放到知识库页面
- 插入官网、博客、内部文档平台

对应最合适的导出格式：

- SVG
- Mermaid

### 场景 3：后续继续编辑

典型用户诉求：

- 未来再打开继续改
- 发给同事继续接着编辑
- 做版本管理或模板沉淀

对应最合适的导出格式：

- ReactFlow JSON
- Mermaid（适合作为轻量工程化表达）

### 场景 4：汇报和表达

典型用户诉求：

- 输出成演示文稿
- 自动生成讲解内容

对应最合适的导出格式：

- PPT
- Slidev
- 讲稿

---

## A.4 建议的导出格式优先级

### P0：强烈建议优先补齐

- PNG
- SVG
- ReactFlow JSON
- Mermaid

原因：

- 业务价值直接
- 用户感知最明显
- 与现有结构化画布能力天然匹配
- 可直接补齐“画完图 -> 带走 -> 再回来”的闭环

### P1：继续强化

- 带元数据的分享包
- 多页面/多视图导出
- 主题化导出（深色/浅色/汇报风）

### P2：延伸能力

- 导出带注释版本
- 导出评审版（含节点说明）
- 导出嵌入包（适配外部页面）

---

## A.5 建议的产品形态

建议不要简单再加一个“导出按钮”，而是升级为统一 `Export Hub`。

### 建议信息架构

#### 一、展示格式

- PNG
- SVG

#### 二、可编辑格式

- ReactFlow JSON
- Mermaid

#### 三、汇报格式

- PPT
- Slidev
- 讲稿

### 建议交互

每个导出项都带一句说明，例如：

- PNG：适合即时分享
- SVG：适合高质量文档和嵌入
- JSON：适合恢复和协同编辑
- Mermaid：适合文档化与 Git 管理

这样可以降低用户理解成本。

---

## A.6 业务价值判断

### 价值一：补齐最短业务闭环

这是当前最接近用户真实需求的一环。

如果用户画完图无法方便导出成常见格式，产品使用链路就会中断。

### 价值二：提升传播效率

很多图不是为了“画出来”，而是为了“传出去”。

导出中心越完善，产品越容易进入真实工作流。

### 价值三：强化资产属性

当图能导出为 JSON / Mermaid 时，图就不再只是截图，而是业务资产。

这为模板库、项目复用、版本管理提供了基础。

### 价值四：为后续协作打基础

统一导出中心本质上也是未来统一分享中心的前置能力。

---

## A.7 实施复杂度与风险

### 复杂度判断

整体属于 **中低到中等复杂度**，是一个适合优先做的方向。

原因：

- 现有导出链路已存在，不是从零起步
- ReactFlow 数据结构已经清晰
- Mermaid 双向能力已具备基础

### 主要风险

#### 风险 1：导出格式之间口径不一致

例如：

- JSON 能保存全部结构
- Mermaid 只能保存部分结构语义
- PNG/SVG 只是展示快照

需要在产品层明确“展示格式”和“可编辑格式”的区别。

#### 风险 2：导出质量差异影响体验

如果 PNG 清晰度低、SVG 样式错乱、Mermaid 丢失信息，会削弱用户信任。

#### 风险 3：导出入口继续膨胀

如果只是在现有菜单上不断堆选项，后续会难以维护。

因此建议直接设计成统一导出中心，而不是临时多加几个按钮。

---

## A.8 推荐结论

建议将“ReactFlow 导出中心”列为近期高优先级业务优化项。

### 推荐 MVP

第一版先做：

- PNG
- SVG
- ReactFlow JSON
- Mermaid

同时保留现有：

- PPT
- Slidev
- 讲稿

### 推荐目标

让用户完成以下闭环：

> 文本/图片生成图 -> 在 ReactFlow 修图 -> 导出为常用格式 -> 后续再次导入继续编辑

这是当前最值得尽快落地的一项。

# 附录 B：ReactFlow / Excalidraw 互转方案评估

## B.1 评估目的

本附录用于评估双画布互转是否值得做、该做到什么深度，以及如何控制范围。

核心问题不是“能不能互转”，而是：

- 什么方向值得做高保真
- 什么方向只适合做降级兼容
- 应该用什么中间模型避免两两耦合

---

## B.2 当前业务问题

当前产品已经拥有：

- ReactFlow：结构化图编辑
- Excalidraw：手绘图/表达型画布

但两者当前更像并列功能，而不是同一套工作流中的两个阶段。

这会带来几个问题：

- 用户在两种画布中重复劳动
- 结构图与演示图无法复用
- 图的“设计态”和“表达态”割裂

---

## B.3 建议的业务角色分工

### ReactFlow 的角色

- 结构化设计
- 布局优化
- 节点边关系表达
- 可计算、可导出、可规范化

### Excalidraw 的角色

- 汇报表达
- 手绘风展示
- 白板脑暴
- 讲解、标注、演示

### 业务意义

这意味着互转不是为了“炫技”，而是为了打通：

> 设计态 -> 表达态  
> 结构图 -> 演示图  
> 白板草图 -> 结构图

---

## B.4 推荐的互转原则

### 原则 1：不追求全量无损

不同画布的设计目标不同，完全无损互转成本非常高，且业务收益未必对等。

更合理的目标是：

- 对“结构化子集”高保真
- 对“自由白板内容”允许降级

### 原则 2：优先做高价值方向

优先级建议：

1. ReactFlow -> Excalidraw
2. Mermaid -> Excalidraw
3. SVG -> Excalidraw
4. Excalidraw -> ReactFlow（仅结构化子集）

### 原则 3：避免两两硬编码映射

如果直接做：

- ReactFlow -> Excalidraw
- Excalidraw -> ReactFlow
- Mermaid -> ReactFlow
- Mermaid -> Excalidraw
- SVG -> ReactFlow
- SVG -> Excalidraw

后续格式越多，耦合会指数上升。

因此建议引入统一的中间文档模型。

---

## B.5 推荐的中间文档模型

建议后续定义统一模型，例如：

`SmartArchitectDocument`

包含以下层次：

### 一、结构层

- nodes
- edges
- hierarchy/group
- viewport

### 二、表现层

- theme
- colors
- font
- stroke/fill
- layout hints

### 三、表达层

- annotations
- hand-drawn mode
- highlights
- presentation metadata

### 四、来源层

- source type
- source fidelity
- import metadata
- conversion warnings

这样后续转换路径变成：

- ReactFlow <-> Document
- Excalidraw <-> Document
- Mermaid <-> Document
- SVG <-> Document（部分高保真，部分视觉导入）

这样可显著降低系统复杂度。

---

## B.6 各转换方向的业务价值评估

### 1. ReactFlow -> Excalidraw

#### 价值

- 结构图一键转演示图
- 技术方案图一键转汇报图
- 适合售前、架构汇报、培训分享

#### 判断

这是最值得优先做的方向之一。

#### 推荐保真目标

- 节点、边、文本、分组尽量保留
- 风格允许转成手绘表达

### 2. Excalidraw -> ReactFlow

#### 价值

- 脑暴草图转结构化图
- 评审草图沉淀成正式图

#### 判断

有价值，但不适合承诺“任意白板图都能完美结构化”。

#### 推荐保真目标

- 只支持结构化子集
- 非结构化自由笔迹、复杂标注仅做忽略或保留为附件

### 3. Mermaid -> Excalidraw

#### 价值

- 文档化图快速转为汇报态
- 工程内的 Mermaid 图可以直接演示化

#### 判断

性价比高，建议优先推进。

### 4. SVG -> Excalidraw

#### 价值

- 外部图形资产导入白板继续演示和批注
- 支持更多第三方图源

#### 判断

很适合作为视觉导入方案，不一定强调结构化还原。

---

## B.7 实施复杂度与风险

### 复杂度判断

属于 **中高复杂度** 项目，但如果分阶段推进，可控。

### 主要风险

#### 风险 1：用户对“互转”抱有无损预期

产品上必须明确：

- 哪些是结构化转换
- 哪些是视觉转换
- 哪些可能产生告警和信息损失

#### 风险 2：格式间语义不一致

例如：

- ReactFlow 支持强结构关系
- Excalidraw 支持自由布局与自由笔迹

两者的能力边界天然不同。

#### 风险 3：直接硬编码会形成长期工程债

如果没有中间模型，后续维护成本会快速升高。

---

## B.8 推荐推进策略

### 第一阶段

先做高价值低风险方向：

- ReactFlow -> Excalidraw
- Mermaid -> Excalidraw

### 第二阶段

再做视觉导入型方向：

- SVG -> Excalidraw

### 第三阶段

最后做结构化子集回转：

- Excalidraw -> ReactFlow（仅子集）

---

## B.9 推荐结论

ReactFlow / Excalidraw 互转值得做，但应按“业务价值优先 + 保真度分层 + 中间模型统一”的方式推进。

### 推荐结论摘要

- 值得立项
- 不建议一次性做全量双向无损
- 建议先打通“结构图 -> 演示图”
- 中长期必须引入统一文档模型

这项能力一旦做通，将显著提升产品差异化。

---

# 附录 C：SVG 引入价值与实现方案评估

## C.1 评估目的

本附录用于评估 SVG 是否应该成为产品中的正式能力，以及应该扮演什么角色。

核心问题不是“要不要支持 SVG”，而是：

- SVG 在产品中是源数据、交换层，还是分发格式
- SVG 引入后能解决哪些业务问题
- SVG 引入是否会增加系统复杂度

---

## C.2 推荐定位

建议将 SVG 定义为：

> **核心分发格式 + 视觉交换层**

而不是唯一源数据。

换句话说：

- 结构化编辑的真实源数据仍然建议保留为内部文档模型 / ReactFlow JSON
- SVG 主要用于展示、分发、嵌入、打印和视觉导入

---

## C.3 SVG 能解决的业务问题

### 问题 1：图片导出质量不够理想

PNG 在缩放、打印、嵌入网页时都不如 SVG。

### 问题 2：外部文档系统嵌入不方便

SVG 非常适合嵌入：

- 文档平台
- 知识库
- 官网页面
- 技术博客

### 问题 3：缺少跨系统图形交换格式

SVG 作为通用图形格式，可作为外部输入输出桥梁。

### 问题 4：讲解和演示图的再加工能力不足

SVG 可作为视觉中间层，导入到 Excalidraw 继续做演示和标注。

---

## C.4 建议的业务使用场景

### 场景 1：高质量导出

将 ReactFlow 或 Excalidraw 图导出为 SVG，用于：

- 方案文档
- 汇报材料
- 网页嵌入
- 打印输出

### 场景 2：网页嵌入

用户可将图嵌入到：

- 企业知识库
- 内部 wiki
- 产品页
- 技术博客

### 场景 3：视觉导入

从第三方图形工具导出的 SVG 可导入系统中，作为：

- Excalidraw 可继续批注的图层
- 后续视觉转化的中间格式

### 场景 4：模板素材沉淀

品牌化图标、业务模块图、行业图例等，可用 SVG 形式沉淀和复用。

---

## C.5 建议的能力边界

### 建议支持

- ReactFlow -> SVG
- Excalidraw -> SVG
- Mermaid -> SVG
- SVG -> Excalidraw（视觉导入）

### 谨慎支持

- SVG -> ReactFlow

原因：

SVG 本质是视觉描述，不天然等于结构化图。

如果强行做 SVG -> ReactFlow，容易出现：

- 结构语义识别不稳定
- 节点边关系错误
- 用户预期过高

因此更合理的策略是：

- 先支持视觉导入
- 后续只对“结构化子集 SVG”做有限识别

---

## C.6 业务价值判断

### 价值一：提升传播品质

SVG 是典型的“高质量传播格式”，这对画图产品很关键。

### 价值二：增强外部系统适配性

一旦支持 SVG，产品更容易进入：

- 文档系统
- 门户系统
- 报告系统
- 内部平台

### 价值三：增强素材复用能力

很多图形资产天然适合以 SVG 形式沉淀。

### 价值四：为 Excalidraw 视觉导入提供桥梁

SVG 可以成为“外部图 -> 白板再表达”的重要桥。

---

## C.7 实施复杂度与风险

### 复杂度判断

如果只做导出和视觉导入，复杂度属于 **中等偏低**。

如果要进一步支持结构化识别，则复杂度会明显上升。

### 主要风险

#### 风险 1：把 SVG 误当作源数据

如果没有清晰边界，后续会出现：

- 一部分能力基于 JSON
- 一部分能力基于 SVG
- 数据一致性难维护

因此必须明确：

SVG 主要是分发和交换层，不是唯一业务主数据。

#### 风险 2：结构化导入预期过高

用户会自然期待“导入 SVG 后可完全编辑成结构图”，但这在通用场景下很难稳定做到。

#### 风险 3：样式与主题不一致

如果不同导出链路的 SVG 风格不统一，会削弱产品专业感。

---

## C.8 推荐推进路径

### 第一阶段：先做导出

- ReactFlow -> SVG
- Excalidraw -> SVG

目标是先解决高质量分发问题。

### 第二阶段：补视觉导入

- SVG -> Excalidraw

目标是把 SVG 变成视觉中间层。

### 第三阶段：探索结构化识别

- SVG -> 内部文档模型 / ReactFlow（仅结构化子集）

目标不是普适支持，而是控制范围做专项能力。

---

## C.9 推荐结论

SVG 值得引入，而且建议正式进入产品能力体系。

### 推荐角色

- 高质量导出格式
- 页面嵌入格式
- 视觉交换层

### 不建议的角色

- 唯一源数据
- 一上来就承诺通用结构化导入

### 推荐结论摘要

- 建议尽快支持 SVG 导出
- 建议与 Excalidraw 的视觉导入联动评估
- 中长期再探索 SVG 的结构化识别边界

---

## 附录总结

这三项专项评估之间并不是孤立的：

- `导出中心` 解决“图怎么带走”
- `ReactFlow / Excalidraw 互转` 解决“图怎么跨场景复用”
- `SVG 引入` 解决“图怎么高质量传播和交换”

如果三者按顺序推进，能够较快形成以下业务闭环：

> 生成图 -> 修图 -> 转演示图 -> 导出/传播 -> 再导入继续编辑

从业务优先级上看，建议排序为：

1. 导出中心
2. SVG 导出与分发
3. ReactFlow -> Excalidraw 互转
4. SVG -> Excalidraw 视觉导入
5. Excalidraw -> ReactFlow 结构化子集回转
