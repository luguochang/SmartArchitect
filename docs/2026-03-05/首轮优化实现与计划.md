# 首轮优化实现记录（2026-03-05）

## 1. 背景与目标

本轮目标聚焦以下问题：

1. 将后端默认大模型配置固定为 right.codes/codex，模型为 gpt-5.2，并兼容历史字段。
2. 将 Flow Canvas / Excalidraw 的“假流式”改为更真实的流式增量渲染。
3. 对前后端接口进行自测驱动优化，降低超时与异常，修复数据兼容问题。
4. 提升图形生成完整度与可读性（尤其 Excalidraw 连线缺失、节点类型异常、画布截断感）。

---

## 2. 本轮已实现内容

### 2.1 模型配置与兼容层（默认配置写死 + 历史字段兼容）

涉及文件：
- `backend/app/core/config.py`
- `backend/app/model_presets.json`
- `backend/app/services/model_presets.py`
- `backend/app/models/schemas.py`
- `backend/app/api/diagnosis.py`
- `backend/app/api/prompter.py`
- `backend/app/api/export.py`
- `backend/app/api/vision.py`

关键实现：

1. 默认配置固定为：
   - provider: `custom`
   - base_url(api_base): `https://www.right.codes/codex/v1`
   - model(model_name): `gpt-5.2`
2. 在 `model_presets.json` 中新增并强制维护默认预设 `default-right-codes`。
3. `ModelPresetsService` 增强：
   - 启动时自动归一化历史字段：`api_base -> base_url`、`model -> model_name`。
   - `get_active_config()` 支持 `api_base` / `model` 兼容入参。
   - 增加 `list_full_runtime_configs()` 与 `get_failover_configs()`，为流式失败切换提供候选配置。
4. Pydantic Schema 层添加别名兼容（AliasChoices）：
   - `base_url` 兼容 `api_base`
   - `model_name` 兼容 `model`
5. 相关 API 端点统一兼容参数：
   - diagnosis / prompter / export / vision 都能接受新旧字段写法。

### 2.2 Flow 生成链路（真流式 + 容错 + 结构规范化）

涉及文件：
- `backend/app/api/chat_generator.py`
- `backend/app/services/chat_generator.py`
- `frontend/lib/store/useArchitectStore.ts`
- `frontend/components/ArchitectCanvas.tsx`

关键实现：

1. 流式失败切换（Failover）：
   - 按候选配置顺序尝试，首 token 超时触发切换。
   - 对上游错误进行分类（限流、鉴权、超时、无效载荷等），便于策略化处理。
2. 更真实的增量输出：
   - 缩短 partial 解析间隔，优先输出可解析的局部节点/边。
   - 增加 token batching，减少 SSE 事件风暴和前端渲染抖动。
3. Partial 数据清洗与归一化：
   - 未知 node type 自动映射到前端支持类型。
   - 对 start/end/decision/task 自动推断 shape。
   - 缺失坐标时给出稳定 fallback 坐标，保证画布可渲染。
4. 前端状态更新节流：
   - generation log 与聊天状态更新做节流，降低高频 setState 压力。
   - 非增量生成前先清空画布，避免历史残留污染。
5. 画布交互稳定性：
   - 生成中抑制自动 `fitView`，结束后再统一适配，避免流式阶段视图抖动。
   - 固化节点/边类型映射常量，兼容 legacy edge type 别名。

---

### 2.3 Excalidraw 链路（流式增强 + 完整性保障）

涉及文件：
- `backend/app/api/excalidraw.py`
- `backend/app/services/excalidraw_generator.py`
- `backend/app/api/vision.py`
- `frontend/lib/store/useArchitectStore.ts`
- `frontend/components/ExcalidrawBoard.tsx`
- `frontend/components/ExcalidrawUploader.tsx`

关键实现：

1. Excalidraw 生成/流式路径增强：
   - 首 token 超时保护。
   - token batching 降低 SSE 频率。
   - partial 解析更早触发，元素更早上板。
2. 连接完整性兜底（非掩盖式质量防线）：
   - 若 AI 输出无有效箭头/连线，基于图形读取顺序自动补最小连接集，避免“孤岛图”。
3. 视觉稳定性改进：
   - 前端 `ExcalidrawBoard` 增加 scene sanitize 与边界校验。
   - 生成完成后 `scrollToContent + fitToViewport`，减少“被截断”的观感。
   - 默认生成尺寸提升到 `1800x1200`，支持更复杂图面。
4. 上传流式体验修复：
   - 逐元素上板，结束事件缺失时仍可安全收尾。
   - 错误信息展示更明确，避免静默失败。
5. Vision 侧稳态优化：
   - 长耗时分析增加心跳 progress 事件，降低前端长时间无响应感。
   - 视觉模型温度和 token 上限调优（更稳定 JSON 输出，降低超长尾时延风险）。

## 3. 本轮关键产出效果（阶段结论）

1. 配置层已实现“写死默认 + 新旧字段兼容”，前后端接口可同时接收 `base_url/model_name` 与 `api_base/model`。
2. Flow 与 Excalidraw 均已具备更真实的流式基础能力，前端可在生成过程中接收更早的 partial 结果。
3. 结构完整性（特别是 Excalidraw 连接线）有了明确质量防线，不再完全依赖模型一次性完美输出。
4. 接口稳定性和错误语义较前一轮更可控，便于继续压缩超时与异常率。

## 4. 下一步计划（第二轮持续优化）

### 4.1 回归与压测闭环（先验证再优化）

1. 重跑 targeted regression：
   - chat stream / excalidraw stream / vision stream-v2 / multipart validation
2. 重跑全量前后端 smoke，产出新版报告并与基线对比。
3. 引入关键时延指标看板：
   - TTFB、首元素时间、完成总时长、超时率、失败率。

### 4.2 真流式体验继续深化（核心）

1. Flow：
   - 进一步降低 partial merge 的等待阈值，确保用户输入后尽快看到首批节点。
   - 优化前端 incremental merge 策略，减少重排导致的“跳动”。
2. Excalidraw：
   - 提升增量解析器对不完整 JSON 的容忍度。
   - 继续优化“流式上板 vs 最终收敛”策略，减少尾段停顿。

### 4.3 复杂度与美观度优化（Prompt + 结构规则双轮驱动）

1. 建立复杂度分级策略（small/medium/large），按场景动态控制节点上限和细节深度。
2. 针对 React Flow 自定义节点库建立生成约束模板（字段规范、节点类型白名单、边类型规则）。
3. 对 Excalidraw 增加布局后处理规则：
   - 防重叠、连线最短路径倾向、关键节点层次保持。

### 4.4 前端可视化验收（Playwright 实测）

1. 每轮优化后固定执行：输入 -> 生成 -> 截图 -> 结构检查。
2. 覆盖两类画布：
   - Flow Canvas：验证首节点出现时间与连通性。
   - Excalidraw：验证首元素出现时间、连接完整性、视口是否完整显示。
3. 形成图像化验收档案（`output/playwright/`）。

### 4.5 目标验收标准（第二轮）

1. 全量接口 smoke：通过率保持 100%。
2. 关键生成链路：
   - 首可视元素时间持续下降。
   - 超时率显著下降。
3. 图面质量：
   - 连通率、节点可读性、画布完整显示稳定达标。
